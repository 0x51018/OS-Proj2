Part 1

이론적 기반: process와 thread의 차이

명세의 핵심 문장은 “process와 thread는 같은 kernel object이고, 무엇을 공유하느냐만 다르다”입니다. xv6에서는 이 말이 특히 중요합니다.

기존 xv6 process

기존 xv6의 struct proc는 실행 문맥과 주소 공간을 한 구조체에 같이 들고 있습니다. 따라서 fork()로 실행 단위를 하나 더 만들면 주소 공간도 함께 복사됩니다.

proc A context kstack pagetable sz
proc B context kstack new pagetable copied sz

이번 과제의 thread

thread도 scheduler 입장에서는 proc입니다. 다만 CLONE_VM으로 생성된 proc는 부모의 mm_struct를 공유합니다.

proc p0contexttrapframe
proc p1contexttrapframe
shared mmpagetable + sz

실행 단위

context, kernel stack, trapframe, PID는 task마다 따로 있습니다. scheduler는 각 thread를 독립적으로 고릅니다.

공유 단위

pagetable과 sz는 mm_struct 안에 있고, thread group이 함께 봅니다.

lifetime 문제

여러 task가 같은 mm을 가리키므로, 마지막 task가 사라질 때만 page table을 해제해야 합니다.

Part 2

명세의 설계: mm_struct, refcount, trapframe slot

1. mm_struct는 주소 공간의 소유권을 표현합니다

기존 xv6에서는 proc가 직접 pagetable과 sz를 들고 있었습니다. skeleton은 이를 mm_struct로 분리했습니다. 이때 refcount는 몇 개의 task가 같은 주소 공간을 공유하는지 나타냅니다. 

struct mm_struct {
  pagetable_t pagetable;
  uint64 sz;
  int refcount;
  struct spinlock lock;
};

refcount 시뮬레이터

mm_struct refcount = 1

마지막 참조가 사라질 때만 page table과 mm이 해제됩니다.

trapframe slot

명세의 NTHREAD=8 때문에 동시에 쓸 수 있는 trapframe slot이 제한됩니다.

2. thread마다 다른 trapframe VA가 필요합니다

thread들은 page table을 공유하므로 모든 thread가 같은 TRAPFRAME VA를 쓰면 register 저장 위치가 충돌합니다. skeleton은 TRAPFRAME 아래에 8개의 slot을 예약하고, 각 thread의 tf_va를 sscratch에 넣어 trap 진입 시 자기 trapframe을 찾도록 바꿨습니다. 

#define NTHREAD 8
#define USERTOP (TRAPFRAME - NTHREAD * PGSIZE)

prepare_return(): w_sscratch(p->tf_va)
uservec:          csrrw a0, sscratch, a0

Part 3

우리가 구현한 코드의 의미

수정의 중심은 kernel/proc.c, kernel/exec.c, user/uthread.c입니다. 아래 코드 지도는 “무엇을 고쳤고 왜 필요한지”를 함수 단위로 정리합니다.

fn == 0을 막지 않은 이유

xv6 user text가 VA 0에서 시작할 수 있어서 정상 함수 주소가 0일 수 있습니다. 그래서 thread_create()와 kclone()에서 function pointer 0을 무조건 오류로 보지 않았습니다.

stack은 kernel이 free하지 않습니다

stack은 thread_create()가 user heap에서 malloc()으로 잡습니다. kernel의 join()은 stack base만 돌려주고, thread_join()이 free()합니다.

fd와 cwd는 공유가 아니라 복제

명세상 thread 생성 시 file descriptor와 cwd는 POSIX thread처럼 공유하지 않고 기존 fork처럼 duplicate합니다. 따라서 같은 파일을 읽어도 file struct는 독립적으로 움직일 수 있습니다.

Expected Questions

예상 질문 및 답변

기출과 유사한 질문을 중심으로, 빈칸/구술/코드 설명 상황에서 바로 말할 수 있게 정리했습니다.

기출 유사 1kill(pid) 했을 때 같은 process의 다른 thread들은 어떻게 되는가?

pid가 leader pid든 일반 thread pid든, target이 속한 thread group 전체가 종료 대상으로 잡힙니다. 구현에서는 먼저 target의 group_leader와 tgid를 찾고, proc table을 돌면서 같은 group에 속한 모든 proc에 killed = 1을 설정합니다. 잠든 thread는 RUNNABLE로 깨워서 killed 플래그를 확인하고 exit 경로로 들어갈 수 있게 합니다.

기출 유사 2find_free_tf_va()의 목적과 내 코드 흐름을 설명하라.

thread들은 같은 page table을 공유하지만 trapframe은 각자 따로 가져야 합니다. 같은 TRAPFRAME VA 하나만 쓰면 여러 thread의 register 저장 공간이 충돌하므로, TRAPFRAME 아래에 있는 여러 slot 중 비어 있는 VA를 찾아야 합니다. find_free_tf_va()는 page table을 확인하면서 유효한 PTE가 없는 slot을 찾고, 그 VA를 새 thread의 tf_va로 돌려줍니다. 이후 mappages()가 그 VA를 새 thread의 physical trapframe page에 연결합니다.

기출 유사 3kexit에서 leader exit 빈칸이 나오면 무엇을 써야 하고 왜인가?

핵심 빈칸은 if(p->group_leader == p) drain_siblings(p); 입니다. 현재 proc가 group leader이면 process 전체가 종료되는 의미이므로 sibling thread들을 먼저 kill하고, ZOMBIE가 될 때까지 기다린 뒤 정리해야 합니다. 반대로 non-leader thread는 자기만 ZOMBIE가 되고, 나중에 join()으로 회수됩니다.

Q4clone(0)과 clone(CLONE_VM)의 차이는?

clone(0)은 fork처럼 새 mm_struct와 새 page table을 만들고 부모 주소 공간을 복사합니다. clone(CLONE_VM)은 새 proc만 만들고 부모의 mm_struct를 공유합니다. 그래서 전자는 부모/자식 메모리가 독립이고, 후자는 heap/global을 공유하는 thread가 됩니다.

Q5왜 mm_get()과 mm_put()이 필요한가?

여러 thread가 같은 mm_struct를 공유하므로 주소 공간의 lifetime을 proc 하나의 lifetime과 분리해야 합니다. mm_get()은 공유 사용자가 늘 때 refcount를 증가시키고, mm_put()은 사용자가 사라질 때 refcount를 감소시킵니다. refcount가 0일 때만 page table과 mm을 free해야 남은 thread가 깨지지 않습니다.

Q6왜 thread마다 trapframe이 따로 필요한가?

trapframe에는 user mode로 돌아가기 위한 register 상태, 예를 들어 epc, sp, a0 등이 저장됩니다. thread마다 실행 위치와 stack, register 값이 다르므로 trapframe을 공유하면 서로의 상태를 덮어써서 잘못된 위치로 복귀하게 됩니다.

Q7새 thread에서 epc, a0, sp를 왜 설정하는가?

epc = fn은 새 thread가 user mode로 돌아갔을 때 fn 함수부터 실행하게 합니다. a0 = arg는 RISC-V calling convention에서 첫 번째 인자를 전달하기 위한 설정입니다. sp = stack + n_pages * PGSIZE는 새 thread가 할당받은 user stack의 맨 위에서 시작하게 하기 위한 설정입니다.

Q8join()과 wait()의 대상은 어떻게 다른가?

join()은 같은 thread group 안의 non-leader ZOMBIE thread를 회수합니다. wait()은 현재 proc의 child 중 group leader인 ZOMBIE process만 회수합니다. thread는 stack 주소를 돌려줘야 하므로 join이 처리하고, process child는 exit status를 돌려줘야 하므로 wait이 처리합니다.

Q9왜 wait()이 thread zombie를 회수하면 안 되는가?

thread stack은 user library가 malloc()으로 할당했기 때문에, 종료 후 stack base를 user에게 돌려주고 free()해야 합니다. 만약 wait()이 thread를 먼저 freeproc() 해버리면 ustack 정보를 잃어버려 stack을 free할 수 없고 memory leak이 생깁니다.

Q10thread_create()와 thread_join()에서 user library가 맡는 책임은?

thread_create()는 user heap에서 stack을 malloc(n_pages * PGSIZE)로 할당하고 clone(..., CLONE_VM)을 호출합니다. thread_join()은 kernel의 join()이 돌려준 stack base를 받아 free()합니다. kernel은 user heap allocator를 모르기 때문에 stack allocation/free는 user library 책임입니다.

Q11왜 non-leader thread의 exec()를 금지했는가?

exec()는 주소 공간 전체를 새 프로그램 image로 바꾸는 syscall입니다. thread들은 같은 page table을 공유하므로 non-leader 하나가 exec를 하면 sibling thread들이 실행하던 code/stack/global이 갑자기 사라질 수 있습니다. 그래서 이 구현에서는 leader만 exec를 허용하고, non-leader는 -1을 반환합니다.

Q12leader가 exec()할 때 왜 commit 직전에 drain_siblings()를 호출하는가?

exec는 파일 열기, ELF 검증, segment load, argument copy 중간에 실패할 수 있습니다. 너무 일찍 sibling을 죽이면 exec 실패 후에도 기존 thread group이 망가집니다. 그래서 새 page table 준비가 성공하고 실제로 p->mm->pagetable을 바꾸기 직전에 sibling들을 정리합니다.

Q13group_leader와 tgid는 각각 무엇인가?

group_leader는 thread group의 대표 proc를 가리키는 포인터입니다. tgid는 thread group id이고 보통 leader의 pid와 같습니다. 각 thread는 서로 다른 pid를 갖지만 같은 group이면 같은 leader와 같은 tgid를 가집니다.

Q14non-leader가 fork()하면 child는 어느 group에 속하는가?

fork는 CLONE_VM 없이 동작하므로 child는 부모의 mm을 공유하지 않고 새 mm_struct와 새 page table을 받습니다. child의 parent는 fork를 호출한 thread이지만, child 자신이 새 thread group의 leader가 되고 tgid = child pid가 됩니다.

Q15freeproc()에서 trapframe mapping을 왜 먼저 unmap하는가?

thread마다 공유 page table 안에 자기 tf_va slot을 mapping해 두었습니다. proc를 정리할 때 이 mapping을 지우지 않으면 slot이 계속 사용 중인 것처럼 남거나 죽은 physical page를 가리킬 수 있습니다. 그래서 uvmunmap()으로 mapping을 제거한 뒤 physical trapframe page를 kfree()합니다.

Q16왜 copyout()을 사용해서 stack 주소나 exit status를 user에게 전달하는가?

kernel은 user pointer를 그냥 C 포인터처럼 직접 대입하면 안 됩니다. user 주소가 현재 page table에서 유효한지 확인하며 안전하게 써야 하므로 copyout()을 사용합니다. join에서는 stack 주소를, wait에서는 exit status를 user memory에 복사합니다.

Q17thread들이 file descriptor와 cwd를 공유하는가?

이 과제의 구현에서는 POSIX thread처럼 fd table을 공유하지 않습니다. kclone()의 common path에서 기존 fork처럼 filedup()과 idup()으로 open file과 cwd를 복제합니다. 그래서 메모리는 공유하지만 fd/cwd는 명세상 duplicate된다고 설명해야 합니다.

Q18왜 NTHREAD 제한이 생기는가?

공유 page table 안에서 thread마다 고유한 trapframe VA slot이 필요하고, skeleton은 TRAPFRAME 아래에 NTHREAD개의 slot만 예약합니다. 모든 slot이 사용 중이면 find_free_tf_va()가 0을 반환하고 새 thread 생성은 실패해야 합니다.

Q19getpid()를 thread 안에서 호출하면 무엇을 반환하는가?

명세상 각 thread도 하나의 proc이고 고유한 pid를 가집니다. 따라서 thread 안에서 getpid()를 호출하면 leader pid가 아니라 그 thread 자신의 pid, 즉 TID에 가까운 값을 반환합니다. group 전체의 id는 tgid로 따로 관리합니다.

Q20이번 과제의 핵심 설계를 한 문장으로 말하면?

proc는 scheduler가 다루는 실행 단위로 남기고, 주소 공간은 mm_struct로 분리해서 여러 proc가 같은 mm을 공유할 수 있게 만든 것입니다. 그래서 clone(0)은 주소 공간 복사, clone(CLONE_VM)은 주소 공간 공유라는 두 의미를 하나의 생성 함수로 처리합니다.
